# 文档标准规范

本文档定义了信息化项目文档的标准规范，包括格式、结构、命名和版本控制等方面的要求。

## 文档格式标准

### 1.1 文件格式

所有文档必须使用 **Markdown (.md)** 格式。

**原因：**
- 易于阅读和编辑
- 支持版本控制
- 可转换为多种输出格式
- 广泛的工具支持

### 1.2 字符编码

所有文档必须使用 **UTF-8** 编码。

### 1.3 行尾符

- Windows: CRLF (`\r\n`)
- Linux/macOS: LF (`\n`)
- 建议在团队中保持一致

### 1.4 缩进与格式

- 使用空格缩进（推荐2或4个空格）
- 禁止使用制表符混合空格
- 行尾不留空格

### 1.5 Frontmatter

每个文档必须包含 YAML frontmatter：

```yaml
---
title: 文档标题
project: {project_name}
version: {version}
status: {status}
author: {author}
date: {date}
updated_at: {updated_at}
---
```

## 文档结构标准

### 2.1 标准结构

每个文档应遵循以下结构：

```
1. Frontmatter (元数据)
2. 标题
3. 引言/背景
4. 正文内容
   4.1 章节一级
       4.1.1 章节二级
           4.1.1.1 章节三级
5. 附录/参考资料
```

### 2.2 章节编号

使用Markdown标题语法：

```markdown
# 一级标题 (文档标题)

## 二级标题 (主要章节)

### 三级标题 (子章节)

#### 四级标题 (详细说明)
```

### 2.3 必需章节

以下章节为文档必需：

| 章节 | 说明 |
|------|------|
| 标题 | 文档标题，简洁明确 |
| 引言 | 文档目的、范围、背景 |
| 正文 | 主要内容 |
| 参考资料 | 引用的标准和文档 |

### 2.4 可选章节

根据需要可添加：

- 术语定义
- 概述
- 约束条件
- 假设
- 风险评估
- 附录

### 2.5 列表格式

**无序列表：**
```markdown
- 项目一
- 项目二
  - 子项目一
  - 子项目二
```

**有序列表：**
```markdown
1. 第一步
2. 第二步
   1. 子步骤一
   2. 子步骤二
```

### 2.6 表格格式

```markdown
| 列一 | 列二 | 列三 |
|------|------|------|
| 数据1 | 数据2 | 数据3 |
```

### 2.7 代码块格式

```markdown
```语言
代码内容
```
```

支持的代码语言：
- `python` - Python代码
- `javascript` - JavaScript代码
- `sql` - SQL语句
- `bash` - Shell脚本
- `json` - JSON数据
- `yaml` - YAML配置

### 2.8 引用格式

```markdown
> 引用内容
```

## 命名规范

### 3.1 文件命名

**格式：** `[文档名称].md`

**规则：**
- 使用中文文档名称
- 不包含特殊字符
- 可选包含版本号：`[文档名称]_v[版本号].md`

**示例：**
- `需求规格说明书.md`
- `需求规格说明书_v1.0.md`
- `测试报告_v1.1.md`

### 3.2 目录命名

阶段目录命名规范：

| 目录 | 说明 |
|------|------|
| 01-立项 | 立项阶段 |
| 02-需求 | 需求阶段 |
| 03-设计 | 设计阶段 |
| 04-开发 | 开发阶段 |
| 05-测试 | 测试阶段 |
| 06-部署 | 部署阶段 |
| 07-管理 | 管理阶段 |

### 3.3 变量命名

**格式：** `{variable_name}`

**规则：**
- 使用小写字母、数字和下划线
- 必须以字母或下划线开头
- 使用下划线分隔单词

**示例：**
- `{project_name}` - 项目名称
- `{company}` - 公司名称
- `{author}` - 作者
- `{start_date}` - 开始日期

### 3.4 标识符命名

**文档ID格式：** `[类别]_[文档类型]`

**示例：**
- `initiation_proposal` - 立项申请书
- `requirements_srs` - 需求规格说明书
- `design_database` - 数据库设计

## 元数据标准

### 4.1 Frontmatter 字段

| 字段 | 类型 | 必需 | 说明 |
|------|------|------|------|
| title | string | 是 | 文档标题 |
| project | string | 是 | 项目名称 |
| version | string | 是 | 文档版本 |
| status | string | 是 | 文档状态 |
| author | string | 是 | 作者 |
| date | string | 是 | 创建日期 |
| updated_at | string | 否 | 最后更新日期 |
| reviewer | string | 否 | 审核人 |
| approver | string | 否 | 批准人 |
| confidentiality | string | 否 | 保密级别 |

### 4.2 状态值

| 状态 | 说明 |
|------|------|
| draft | 草稿 |
| review | 审核中 |
| approved | 已批准 |
| published | 已发布 |

### 4.3 保密级别

| 级别 | 说明 |
|------|------|
| public | 公开 |
| internal | 内部 |
| confidential | 机密 |
| secret | 绝密 |

### 4.4 版本号格式

采用语义化版本号：`MAJOR.MINOR.PATCH`

- **MAJOR**: 重大变更
- **MINOR**: 新增功能
- **PATCH**: 错误修复

**示例：**
- `1.0.0` - 初始版本
- `1.1.0` - 新增功能
- `1.1.1` - 修复错误
- `2.0.0` - 重大更新

## 版本控制规范

### 5.1 版本控制工具

推荐使用 Git 进行版本控制。

### 5.2 提交信息格式

```
<类型>: <简短描述>

<详细描述>

<关联问题>
```

**类型：**
- `feat`: 新功能
- `fix`: 修复
- `docs`: 文档
- `style`: 格式
- `refactor`: 重构
- `test`: 测试
- `chore`: 构建

**示例：**
```
docs: 更新需求规格说明书

添加新的用户故事章节，更新版本号为1.1.0

Closes #123
```

### 5.3 分支策略

```
main (主分支)
  └─ develop (开发分支)
       ├─ feature/xxx (功能分支)
       ├─ hotfix/xxx (修复分支)
       └─ release/xxx (发布分支)
```

### 5.4 标签规范

发布版本时创建标签：

```bash
git tag -a v1.0.0 -m "发布 v1.0.0"
git push origin v1.0.0
```

### 5.5 变更日志

维护 `CHANGELOG.md` 文件记录变更：

```markdown
# 变更日志

## [1.0.0] - 2024-01-15

### 新增
- 初始版本发布

### 修复
- 修复模板变量替换问题
```

## 质量标准

### 6.1 内容质量

- **准确性**: 信息准确无误
- **完整性**: 内容完整无遗漏
- **清晰性**: 表达清晰易懂
- **一致性**: 风格和术语一致

### 6.2 格式质量

- **规范命名**: 遵循命名规范
- **结构完整**: 包含必需章节
- **格式统一**: 使用统一格式
- **语法正确**: Markdown语法正确

### 6.3 元数据质量

- **完整填写**: 所有必需字段已填写
- **准确更新**: 更新日期及时
- **状态同步**: 状态与实际情况一致

### 6.4 审核标准

文档审核前应满足：

- [ ] 所有变量已填充
- [ ] 内容完整无遗漏
- [ ] 格式规范统一
- [ ] 无拼写和语法错误
- [ ] 元数据完整准确
- [ ] 参考资料有效

### 6.5 验证检查

使用 `validate_template.py` 验证：

```bash
python scripts/validate_template.py document.md --fix
```

### 6.6 质量评分

文档质量评分表：

| 项目 | 满分 | 说明 |
|------|------|------|
| 内容完整性 | 25 | 所有必需内容是否完整 |
| 内容准确性 | 25 | 信息是否准确无误 |
| 格式规范性 | 20 | 是否符合格式规范 |
| 元数据完整性 | 15 | 元数据是否完整 |
| 参考资料有效性 | 10 | 参考资料是否有效 |
| 整体可读性 | 5 | 文档是否易于阅读 |

**评分标准：**
- 90-100分：优秀
- 80-89分：良好
- 70-79分：合格
- 60-69分：需改进
- <60分：不合格
